阶段二 · Schedule / 07

7 · Scheduler：安排任务执行

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
先分清三者：Lane 选更新，Scheduler 排时间，Fiber 做计算

Lane 标记本轮要处理哪些 update；Scheduler 为整个 root 安排一次执行机会；轮到这个任务后，Reconciler 才沿 Fiber 树执行 beginWork/completeWork。

最容易误解的一点：Scheduler 不认识组件，也不会给每个 Fiber 建一个任务。它调度的是“继续处理某个 root”这个回调；Fiber 是回调内部逐个处理的工作单元。

先固定一个场景：输入框 + 慢列表

function SearchPage() {
  const [text, setText] = useState('');
  const [query, setQuery] = useState('');

  function onChange(e) {
    setText(e.target.value);              // 紧急：输入框立刻显示
    startTransition(() => {
      setQuery(e.target.value);           // 可延后：重新计算大列表
    });
  }
}

假设列表很大，Transition Render 需要多个时间片。下面所有概念都围绕这件事讲：

① Transition update 入队
setQuery 的 update 带 Transition Lane；root.pendingLanes 记录它。
② React 为 root 安排一个低优任务
ensureRootIsScheduled 把“处理这个 root”的回调交给 Scheduler。
③ Scheduler 执行回调，Fiber Render 开始
React 在 workInProgress 树上逐个计算列表 Fiber，真实 DOM 尚未改变。
④ 时间片用完，React 暂停
workInProgress 指向“下一个还没处理的 Fiber”，Scheduler 安排后续回调。
⑤ 浏览器处理下一次输入
setText 产生 Sync Lane。React 发现它比正在处理的 Transition 更急。
⑥ 先完成 Sync，再重新处理 Transition
低优回调被替换；高优更新先 Render 和 Commit。Transition update 没有消失，随后基于最新 current 重新计算列表。
先记住结果：输入框先更新，慢列表稍后更新。Scheduler 的全部工作，就是让第 ⑤ 步产生的紧急更新不必排在漫长的列表 Render 后面。

Scheduler 实际收到的是什么

Scheduler 只接收“回调函数 + Scheduler 优先级”。它不认识 Lane、root 或 Fiber。React 把 root 放进回调的闭包中，再把这个回调交给 Scheduler：

const callback = performConcurrentWorkOnRoot.bind(null, root);
const task = scheduleCallback(NormalPriority, callback);

// Scheduler 内部看到的近似任务
task = {
  callback,          // 执行它，才会回到 React 并处理这个 root
  priorityLevel,
  startTime,
  expirationTime
};

📖 逐块拆解上面这段代码

上面代码涉及 6 个东西，一个个认识：

① performConcurrentWorkOnRoot(root, didTimeout) —— React 的「干活总入口」

React 里干实事的那个函数。一被调用就开始处理这个 root（走完 render → commit）。类比：像 Redux 的 store.dispatch，一调用触发一整套流程。

它需要 2 个参数：root（要处理哪棵树）和 didTimeout（是不是超时要强制同步做完）。

② .bind(null, root) —— 把 root 提前塞进函数

bind 是所有函数都有的方法，作用：预填参数，返回一个新函数。

performConcurrentWorkOnRoot.bind(null, root)
//                            ↑     ↑
//                        不绑 this  预填 root

得到的 callback 是一个只需要 1 个参数（didTimeout）的新函数，root 已经焊在里面。类比：onClick={() => handleDelete(item.id)} —— item.id 焊进闭包，下游不用知道它。

③ scheduleCallback(priority, callback) —— Scheduler 的「排队入口」

Scheduler 包对外的核心 API。把回调登记到 Scheduler 内部的最小堆，返回一个 task 对象。

// 类比 setTimeout,但多了个优先级
setTimeout(callback, 100);                       // 到点调
scheduleCallback(NormalPriority, callback);       // 到点+时间片可用时调

④ NormalPriority —— Scheduler 的优先级常量（不是 Lane！）

Scheduler 有它自己的一套优先级系统，只有 5 个级别，比 Lane 简单得多：

ImmediatePriority    = 1   // 立刻执行
UserBlockingPriority = 2   // 用户交互
NormalPriority       = 3   // 一般
LowPriority          = 4   // 低
IdlePriority         = 5   // 空闲

Lane 是 React 的语言，Scheduler 优先级是 Scheduler 的语言，中间有一层翻译：

lanesToSchedulerPriority(lanes)   // Lane → Scheduler 优先级

⑤ task —— Scheduler 内部的任务对象

scheduleCallback 返回一个 task 对象，是 Scheduler 为这次调度创建的记账条目。React 拿到后存在 root.callbackNode，后面可能用它做：

  • 取消：cancelCallback(task) —— 有更高优来了，取消旧的
  • 对比：判断已排的任务是不是需要重排

类比 setTimeout 返回的 id：const id = setTimeout(fn, 100); clearTimeout(id); —— task 就是「id」的升级版（不只是数字，是个完整对象）。

⑥ task 上的四个字段

字段是什么类比 setTimeout
callback到点要跑的函数（就是 bind 出来那个）setTimeout 的第一个参数
priorityLevelScheduler 优先级决定在最小堆里的排位
startTime什么时候可以开始跑（一般是 now()）让 task 提前入堆但延迟启动
expirationTime什么时候算超时要马上做兜底：低优等太久也保证被执行

📝 逐句翻译成人话

const callback = performConcurrentWorkOnRoot.bind(null, root);
// 把"干这个 root 的活"打包好,root 焊进去,
// 得到一个能直接 callback() 调的函数

const task = scheduleCallback(NormalPriority, callback);
// 告诉 Scheduler:"这函数用 Normal 优先级排队,到点你调用它"
// Scheduler 建了个 task 条目扔进最小堆,把 task 返回给我

task = {
  callback,           // ← 刚才 bind 那个,到点会被调
  priorityLevel,      // ← NormalPriority
  startTime,          // ← 现在
  expirationTime      // ← now + 5000ms 之类,超过就得强制做完
};
Scheduler 眼里的世界只有 task 这个对象。它不认识 root、Fiber、Lane，就是个「带优先级的 setTimeout」。到时机时调 task.callback()，后面所有事都在 callback 里的 React 世界发生。

假设 App 中有一万个 Fiber，React 也不会提交一万个 Scheduler 任务。从 React 的角度看，通常是为一个 root 保留一个并发回调；回调真正执行后，Reconciler 才在内部逐个处理 Fiber。

root.callbackNode      // 当前并发 Scheduler 任务；同步路径可能为 null
root.callbackPriority  // root 当前已安排工作的 Lane 优先级

新的 update 到来后，ensureRootIsScheduled 会再次检查 root：

  • 还没有工作入口：根据 Lane 安排同步入口或并发 Scheduler 任务。
  • 已有同等优先级任务：继续复用，不重复安排。
  • 来了更高优更新：取消旧的并发回调，按更高优先级重新安排。
  • 已经没有待处理 Lane：取消回调并清空 callbackNode。
// ensureRootIsScheduled 的核心判断（简化）
nextLanes = getNextLanes(root.pendingLanes);
newPriority = getHighestPriorityLane(nextLanes);

if (newPriority === root.callbackPriority) return; // 已安排同等工作
if (root.callbackNode !== null) cancelCallback(root.callbackNode);

if (newPriority === SyncLane) {
  scheduleSyncCallback(performSyncWorkOnRoot);    // 同步队列由微任务尽快刷新
  root.callbackNode = null;
} else {
  root.callbackNode = scheduleCallback(
    lanesToSchedulerPriority(nextLanes),
    performConcurrentWorkOnRoot.bind(null, root)
  );
}
  root.callbackPriority = newPriority;
为什么 Sync 能明显插队？Transition 通常等待 Scheduler 的并发回调，并且工作循环会让位；Sync 更新改走同步刷新入口。当前 JavaScript 任务结束进入微任务检查后，React 会尽快完成 Sync Render/Commit。

🔀 澄清：Sync / Concurrent 在 React 里到底指什么

Sync 和 Concurrent 这两个词在 React 里其实指代 3 层不同的东西，容易混：

层次Sync 是啥Concurrent 是啥
① 应用模式（老/新）ReactDOM.render（React 17 Legacy）createRoot（React 18+）
② Lane 优先级SyncLane（最高优 bit）TransitionLane、DefaultLane 等其它 lane
③ 渲染路径（这段代码在讲的）Sync 路径：微任务刷新Concurrent 路径：Scheduler 时间切片

关键：用了 createRoot 不代表所有更新都「并发」 —— SyncLane 依然走同步路径。同一个 app 里两条路径并存，看每个 update 的 lane 走哪条。

两条渲染路径的核心对比（层次 ③）

前端类比

  • Sync 路径 ≈ queueMicrotask(fn) —— 当前同步代码一结束就跑，浏览器还没来得及 paint，微任务里也不让位
  • Concurrent 路径 ≈ requestIdleCallback + 手动 5ms 让位 —— 排到宏任务队列里等，跑起来还会让位给浏览器 paint / 用户输入

具体：哪种 setState 走哪条路

场景lane路径
离散事件回调（click / keydown / input）里的 setStateSyncLaneSync

时间线：两条路径怎么共存

t=0   startTransition(() => setBigList)
      → lane = TransitionLane
      → Concurrent 路径:排 Scheduler task

t=1   Scheduler 分片跑 concurrent render(慢慢渲大列表)

t=2   用户点按钮 setX(1)
      → lane = SyncLane
      → Sync 路径:
        • cancelCallback(旧 concurrent task) ★取消低优
        • scheduleSyncCallback → queueMicrotask

t=3   当前 click 回调结束
      → 微任务立刻跑 → performSyncWorkOnRoot
      → workLoopSync 一口气跑完 → commit(Sync 上屏)

t=4   Sync commit 后 ensureRootIsScheduled 又看一眼
      → pendingLanes 还有 TransitionLane
      → 重排 concurrent task

t=5   Scheduler 继续跑 concurrent render → commit(Transition 上屏)

同一个 app 里，两条路径可以在几毫秒内交错触发。

常见误解

误解事实
React 18 所有更新都是并发的❌ SyncLane 还是走同步路径，跟 17 类似
concurrent 比 sync 快❌ concurrent 单次渲染反而更慢（有分片开销），换来的是「不阻塞交互」
同步 = 老版本、并发 = 新版本❌ 18 里两条路并存，看 lane 分别走
flushSync 让 concurrent 变 sync✅ 对，临时切到 Sync 路径立刻刷
一句话记忆：Sync 路径 = 「微任务里立刻做完，不让位不被打断」；Concurrent 路径 = 「宏任务里排队做，5ms 让一次位，随时可能被高优抢占」。用户交互走 Sync 保证跟手；startTransition、大列表更新走 Concurrent 保证不卡输入。两条路径各司其职、并存于同一 app。
cancelCallback 取消的只是 Scheduler 任务。它不会删除 update，也不会回滚 DOM，因为低优 Render 根本还没有进入 Commit。update 仍在 Hook/Class 的更新队列中，对应 Lane 仍留在 root.pendingLanes。

抢占发生时，内存里每样东西去了哪里

先区分四样东西，它们不是同一个对象：

东西暂停时存在哪里高优更新到来后
低优 update组件 Hook/Class 的 update 队列保留，之后仍要处理
低优 Laneroot.pendingLanes保留，表示 root 仍有低优工作
已经算了一半的 Fibercurrent 对应的 alternate 对象，也就是部分完成的 WIP 树如果改为渲染新 Lane，这批半成品可以被覆盖并重算
下一个计算位置模块变量 workInProgress同 Lane 续算时保留；切换 Lane 重算时重新指向新的 root WIP
用户正在看的界面root.current 对应的 current 树和真实 DOM完全不变，因为低优结果尚未 Commit
// Transition 时间片暂停时，可以把现场理解成：
workInProgressRoot = root;                 // 正在处理哪棵树
workInProgressRootRenderLanes = TransitionLane; // 正在处理哪批更新
workInProgress = listItem301Fiber;         // 下次从哪个 Fiber 继续

root.current = currentTree;                // 已提交树，始终可用
root.finishedWork = null;                  // 还没算完整，不能 Commit

情况 A：只是时间片结束，没有更高优更新

workInProgressRootRenderLanes 仍是 Transition，workInProgress 仍指向列表中尚未处理的 Fiber。下一次 Scheduler 回调到来后，React 继续使用同一棵 WIP，从该 Fiber 接着算。这叫暂停和续算，不是抢占。

情况 B：暂停期间来了 Sync 更新

  1. 浏览器处理输入事件，setText 创建 Sync update。
  2. Sync Lane 加入 root.pendingLanes，此时 root 同时有 Sync | Transition。
  3. ensureRootIsScheduled 发现现有低优回调不够急，调用 cancelCallback 取消它；Sync 更新随后进入同步刷新路径。
  4. 高优回调运行时，getNextLanes 选出 Sync。它与当前正在构建的 Transition lanes 不同。
  5. prepareFreshStack(root, SyncLane) 从仍然完好的 root.current 创建新的根 WIP，并重置 workInProgress。先前那棵低优 WIP 不会提交。
  6. Sync Render 遍历更新队列：执行 Sync update，暂时跳过 Transition update，并把后者保留到基础队列。
  7. Sync Commit 后，输入框更新。Transition Lane/update 仍然 pending，于是 React 再安排低优任务，基于最新 current 重算列表。
“丢弃低优 WIP”丢的是计算结果，不是更新请求。半成品 Fiber 上算出的 props/state/flags 没有提交，可以覆盖；原始 Transition update 仍保存在队列中，所以稍后能够重新算出正确结果。

🎨 一次串起来：内存里 5 样东西怎么互动

上面的表格和 A/B 两种情况密度大，配一张关系图和画画类比就通了。先看 5 样东西的关系：

                     内存中同时存在的东西
   ┌─────────────────────────────────────────────────────┐
   │                                                       │
   │   ┌──────────────┐          ┌──────────────┐         │
   │   │  current 树  │◄────────►│    WIP 树    │         │
   │   │  (屏幕对应)  │ alternate│  (正在算的) │         │
   │   └──────┬───────┘          └──────┬───────┘         │
   │          │                          │                 │
   │    root.current           workInProgress             │
   │    (屏幕就是它)          (指针:下一个算哪)          │
   │          │                                            │
   │          ▼                                            │
   │       真实 DOM ◄─── 用户实际看到的                    │
   │                                                       │
   │   ─────────────────────────────────────────           │
   │                                                       │
   │   root.pendingLanes  = 全局待办 lane 集合              │
   │                                                       │
   │   每个组件 fiber 上:                                   │
   │     updateQueue.shared.pending = update 环(未消耗)     │
   │     updateQueue.baseQueue      = 上轮跳过的 update     │
   └─────────────────────────────────────────────────────┘

关键：current 和 WIP 不是两棵独立的树，通过 alternate 字段互相指向，共享大部分节点 —— 只是双缓存的两个视角。

画画类比：一秒分清 5 样东西

想象你是画家：

React 概念画画类比
current 树挂墙上的成品画（用户已经看到的）
WIP 树画室里的草稿（正在改的）
workInProgress 指针画笔当前落在纸上哪一点
update 队列客户给的需求笔记（帽子改红色、背景加月亮...）
WIP 上算过的 props/state/flags草稿上已经上过色的部分
真实 DOM墙上那张成品，客户能看到

创作流程：看着墙上的成品（current）→ 拿新纸（WIP）对着笔记（update）一笔笔画 → 画完摘下旧画挂新画（commit）。

场景 A 具体走一遍：时间片结束（暂停，不是抢占）

你画到主角的鼻子，累了要休息：

草稿(WIP):              已画完头发、眼睛,正在画鼻子   ← 半成品保留
画笔位置(workInProgress): 停在鼻子右边                   ← 保留
挂墙上的成品(current):    老样子                         ← 保留
客户笔记(update 队列):    全部还在                       ← 保留

休息完继续从鼻子右边接着画，半成品草稿没动过 —— 下次用同一张纸。没人受损，一切原样保留。

场景 B 具体走一遍：Sync 抢占（真·丢弃）

你画到鼻子时，客户跑进来：「先给墙上那张加副眼镜（Sync）！」 —— 你手上这张草稿其实用不上了。

步骤 1：取消当前的画

cancelCallback(旧 concurrent task);   // Scheduler 别再叫我继续画

步骤 2：prepareFreshStack(root, SyncLane) —— 准备新草稿

workInProgress = createWorkInProgress(root.current, null);
workInProgressRootRenderLanes = SyncLane;

关键理解：createWorkInProgress 不是真的丢一张纸再拿一张新的。它做的是：

  1. 复用 current.alternate（就是"画了一半的鼻子"那个 fiber 对象）
  2. 重置上面的关键字段：memoizedState、flags、lanes、childLanes —— 把"画过的痕迹擦掉"
  3. 让 workInProgress 指针重新指向 root（画笔挪回起点）

丢弃的是什么？fiber 节点上"算了一半的计算结果"（memoizedState / flags 等）。fiber 对象本身复用了，节省 GC。

步骤 3：Sync render

从 root 从头遍历，处理每个 fiber 的 update 队列：

  • 遇到 Sync update → 执行
  • 遇到 Transition update → 跳过，存进 baseQueue

⚠️ 注意：Transition update 没有丢！它只是被跳过，进了 baseQueue 等下轮。

步骤 4：Sync commit → 上屏

真实 DOM 更新：加眼镜完成。用户看到了。

步骤 5：Sync 完了，Transition 还没干

root.pendingLanes 现在只剩 TransitionLane
ensureRootIsScheduled 又跑一遍 → 排新的 concurrent task

步骤 6：Scheduler 重新跑 Transition

prepareFreshStack(root, TransitionLane) → workInProgress 又回 root → 从头 render。这次 List 组件 UpdateQueue 里，baseQueue 那个 setItems 还在，重新执行 → commit。

"丢弃低优 WIP" 三个关键问答

问答
Q1：丢的是 update 吗？不是。update 全在 fiber.updateQueue 里保留。丢的是"上一轮处理 update 后算出的中间结果"
Q2：DOM 会变吗？不会。半成品 WIP 从来没进入 commit，DOM 只在 commit 时被修改
Q3：数据会丢吗？不会。update 是"操作意图"（"我想把 count 加 1"）不是"结果"（"count 现在是 5"），重跑能算出相同结果

React 所有并发保证都建立在这个约束上：中间状态永远不上屏，DOM 只在完整 commit 时才变。

Scheduler 回调怎样驱动 Fiber 工作循环

Scheduler 调用 root 回调后，控制权回到 React Reconciler。Reconciler 准备本次 renderLanes，然后重复处理 Fiber：

function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress); // 处理一个 Fiber
  }
}

每个 Fiber 是一个可暂停边界。处理完一个 Fiber 后，React 检查 shouldYield()：

  • 还有时间：继续处理下一个 Fiber。
  • 时间片用完：退出工作循环，保留 workInProgress 指针。
  • Scheduler 再次给 root 执行机会：从这个指针继续。
Fiber 与时间切片的真正关系：Scheduler 只说“现在该暂停了”；Fiber 链表结构让 React 知道暂停在哪里、下次从哪里继续。没有 Fiber 这种工作单元，Scheduler 即使要求让位，React 也很难保存一半的树计算进度。

🎬 展开：Scheduler 和 Fiber 的两层分工

上面这段的核心是两层控制反转——Scheduler 掌握「何时停」，Fiber 掌握「停在哪、怎么续」。分开讲。

workLoopConcurrent 每一句拆开

function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }
}
  • workInProgress：模块级变量，存"下一个要处理的 fiber"。上次让位时它保留了位置，这次进循环从它开始 —— 它就是 Fiber 世界里的"书签"。
  • shouldYield()：Scheduler 提供的"该停了吗"检查。时间用光或有更急任务，返回 true。谁在调它？Reconciler。Scheduler 只负责回答，不主动打断。
  • performUnitOfWork：处理一个 fiber，不是整棵子树。处理完修改 workInProgress 指针指向下一个。

performUnitOfWork 内部：这才是「可暂停」的关键

function performUnitOfWork(unitOfWork){
  const next = beginWork(unitOfWork.alternate, unitOfWork, renderLanes);
  //           ↑ 处理这个 fiber,有子节点返回 child,否则返回 null
  if (next === null) {
    completeUnitOfWork(unitOfWork);   // 无子,回归(找 sibling 或往上)
  } else {
    workInProgress = next;              // 有子,深入
  }
}

遍历顺序（举例）：

         A
        / \
       B   E
      / \
     C   D

访问顺序:
  begin A → begin B → begin C → complete C
→ begin D → complete D → complete B
→ begin E → complete E → complete A

每次访问一个 fiber 就是"一个工作单元"（一次 performUnitOfWork）。每个工作单元之间都是可让位点。

递归 vs 迭代：Fiber 为什么天然「可暂停」

React 15（Stack Reconciler）—— 递归：

function render(fiber){
  process(fiber);
  for (const child of fiber.children){
    render(child);   // ← 一进递归就出不来
  }
}
render(root);   // 一口气跑完,期间无法让位

调用栈是 JS 引擎的黑盒：你看不见递归到哪一层、没法保存栈上的进度、只能一口气跑完。

React 16+（Fiber）—— 迭代：

let workInProgress = root;   // ← 关键:一个"看得见"的指针
while (workInProgress !== null && !shouldYield()){
  performUnitOfWork(workInProgress);  // 内部会移动指针
}
// 循环 break 出来时,workInProgress 保留在最后位置
核心差别：递归的进度存在调用栈里，JS 引擎管，你动不了。迭代的进度存在一个变量里，你自己管，随时可保存/恢复。Fiber 就是把「树的深度优先遍历」从递归改成迭代的产物——为此每个节点必须有 child / sibling / return 三个指针帮迭代找路。

前端类比：像 generator 或分批处理

类比 1：generator 函数

function* traverseTree(root){
  yield root;                        // ★让位点
  for (const child of root.children){
    yield* traverseTree(child);
  }
}

const iter = traverseTree(fiberTree);
while (!shouldYield()){
  const {value, done} = iter.next();  // 每次拿一个
  processFiber(value);
}

📘 generator 速成：大多数前端没写过，先补一下

普通函数一进去必须跑完；function* 带星号的 generator 可以「暂停」。

function* foo(){
  console.log(1);
  yield;           // ★暂停点
  console.log(2);
  yield;
  console.log(3);
}

const gen = foo();    // ⚠️ 调用不会立刻执行,只是拿到"迭代器"
gen.next();  // 打印 1,停在第一个 yield
gen.next();  // 从暂停点恢复,打印 2,停在第二个
gen.next();  // 打印 3,函数结束

神奇之处：函数被「冻结」在 yield 那行，下次 .next() 从那行继续。JS 引擎帮你自动保存函数内部所有状态（局部变量、for 走到哪、if 进哪个分支）。

yield 还能带值，.next() 返回 {value, done}：

function* counter(){ yield 'a'; yield 'b'; yield 'c'; }
const iter = counter();
iter.next();  // { value: 'a', done: false }
iter.next();  // { value: 'b', done: false }
iter.next();  // { value: 'c', done: false }
iter.next();  // { value: undefined, done: true }

yield* 委托给另一个 generator（把它的 yield 都接力出去）：

function* inner(){ yield 'a'; yield 'b'; }
function* outer(){
  yield 1;
  yield* inner();   // ← 把 inner 的 yield 都在这里接力出去
  yield 2;
}
[...outer()]   // [1, 'a', 'b', 2]

这就是递归 generator 的核心 —— 上面 traverseTree 里就是靠 yield* 递归遍历子树的。

🌳 具体走一遍 traverseTree(A)

       A
      / \
     B   E
    / \
   C   D
第 1 次 next():  yield A                                       ↩ A
第 2 次 next():  进 A 的 for, yield* traverseTree(B) → yield B     ↩ B
第 3 次 next():  进 B 的 for, yield* traverseTree(C) → yield C     ↩ C
第 4 次 next():  C 无子结束,回 B 的 for, yield* traverseTree(D)    ↩ D
第 5 次 next():  D 结束,回 A 的 for, yield* traverseTree(E)        ↩ E
第 6 次 next():  全部结束                                          ↩ {done: true}

顺序：A → B → C → D → E —— 就是深度优先遍历。

🔑 神奇之处：break 出 while 后，迭代器状态还在

const iter = traverseTree(fiberTree);   // ← 拿到迭代器,还没开始
while (!shouldYield()){
  const {value, done} = iter.next();
  if (done) break;
  processFiber(value);
}
// 5ms 用光 → shouldYield() 返 true → break 出 while
// 但 iter 变量还在!它内部保存着"下次从哪个 yield 继续"

// 下一个时间片:
while (!shouldYield()){
  const {value, done} = iter.next();   // ← 从上次暂停点继续!
  ...
}

generator 自动帮你保存「遍历到哪了」这个状态。什么都不用干，直接 iter.next() 就从原地续。

🔀 映射到 Fiber

generator 版本Fiber 版本
yield 暂停点每个 fiber 处理完的位置
迭代器内部状态（JS 引擎自动管）workInProgress 指针（手动管）
iter.next()下次循环从 workInProgress 开始
yield* 递归child/sibling/return 三指针遍历

为什么 React 不直接用 generator？（1）每次 .next() 有上下文切换开销，累积起来慢；（2）generator 遍历路径一旦生成就固定，Fiber 可以抢占时随意改指针；（3）早期兼容性。但功能等价 —— 理解 generator 就理解了 Fiber 遍历的心智模型。

回到主线：Fiber 手动实现了 generator 的效果 —— 用 workInProgress 指针代替 yield，用 beginWork/completeWork 代替 yield 之间的代码。

类比 2：分批处理大数组（你可能写过）

let index = 0;
function processChunk(){
  const start = performance.now();
  while (index < items.length && performance.now() - start < 5){
    process(items[index]);
    index++;                          // ← 保存进度
  }
  if (index < items.length){
    setTimeout(processChunk, 0);      // ← 让位,下轮继续
  }
}

对照关系：index = workInProgress；performance.now() - start < 5 = !shouldYield()；setTimeout(processChunk, 0) = Scheduler 排下一片；items[index] = fiber 树下一个节点。你其实早就写过这个模式了，只是 React 把它做得更精细。

暂停后怎么恢复：Scheduler 续作机制

关键问题：workLoopConcurrent break 出来后，Scheduler 怎么知道要回来？

function performConcurrentWorkOnRoot(root, didTimeout){
  const lanes = getNextLanes(root);
  const status = renderRootConcurrent(root, lanes);
  //             ↑ 里面调 workLoopConcurrent

  if (status === RootInProgress) {
    // 还没渲完(时间片打断),返回自己作为续作
    return performConcurrentWorkOnRoot.bind(null, root);   // ★
  }
  commitRoot(root);   // 渲完了才提交
}

流程：

Scheduler 分了 5ms 给你
  → 调 callback = performConcurrentWorkOnRoot
  → callback 里调 workLoopConcurrent
  → workLoop 处理了 30 个 fiber, shouldYield()=true, break
  → workInProgress 保留在第 31 个 fiber
  → performConcurrentWorkOnRoot 判断"没渲完",返回自己
  → Scheduler 收到"续作",重新入堆

Scheduler 下一片又轮到我
  → 又调 callback (还是 performConcurrentWorkOnRoot)
  → callback 里 workLoopConcurrent 从 workInProgress 继续
  → 处理第 31, 32, 33... 个 fiber

Scheduler 只做两件事：分片 + 再叫我。具体从哪里继续，是 React 内部靠 workInProgress 记住的。

如果没有 Fiber 会怎样（反证）

function render(fiber){
  process(fiber);
  for (const child of fiber.children){
    if (shouldYield()) return ???;   // ← 让位怎么办?
    render(child);
  }
}

问题：

  • 我 return 了，父级递归怎么知道该继续还是终止？
  • 就算它知道要"记住位置"，怎么记？调用栈里还有 100 层等着回归呢
  • 下次怎么"从中间某一层"继续？没辙

只能一口气跑完。所以 React 15 时代想让位是架构上做不到——不是不想，而是不能。Fiber 把"隐式的调用栈"变成"显式的链表"：显式 = 我能看、能存、能改。让位就变成"把指针留原地"，恢复就变成"从指针继续"。

Scheduler / Reconciler / Fiber 三方交互图

Scheduler                           React Reconciler                Fiber 树

    │                                │                                │
    │ 调 callback()                  │                                │

一句话记忆：
  • Scheduler 说「该停了」（shouldYield()）
  • Fiber 结构让「能停」&「能续」（workInProgress + 链表指针）
  • 两者缺一不可：光有 Scheduler 没 Fiber = 递归里想让位也让不了；光有 Fiber 没 Scheduler = 无人告诉何时让位

Scheduler 是「何时」，Fiber 是「如何」—— 完美分工。

到底是谁可打断：答案是并发 Render 工作循环

阶段能否时间切片暂停原因
Scheduler 排队/选择任务不叫“React Render 被打断”它只决定何时调用回调，并提供 shouldYield
Concurrent Render可以只修改内存中的 WIP Fiber；每完成一个 Fiber 后可退出循环
Sync Render不会按时间片让位同步工作循环会一直算到完成
Commit不可以正在修改 DOM、ref 和执行 Layout effect，中途停会暴露不一致界面

发生暂停的直接条件：当前是并发 Render，workInProgress !== null，并且处理完一个 Fiber 后 shouldYield() 返回 true。React 退出 workLoopConcurrent，把控制权交回浏览器。

高优更新何时能进来：JavaScript 正在运行时，点击事件不能同时插入执行。必须先由低优 Render 在时间片边界让位，浏览器才有机会派发点击；点击产生高优 update 后，React 下一次选择 lanes 时才发生抢占。

恢复：Lane 没变

保留 workInProgress 指针和部分 WIP 树；下一次回调从原位置继续。

重算：来了更高优 Lane

prepareFreshStack 从 current 建立新 WIP；旧半成品不提交。原 update 仍在队列中，之后重新处理。

Lane 怎样转换成 Scheduler 优先级

Lane 的分类比 Scheduler 更细。ensureRootIsScheduled 先选出 root 下一批要处理的 Lane，再把它映射为 Scheduler 能理解的任务优先级：

React 中的更新Lane 负责Scheduler 负责
点击、键盘等紧急更新选出高优 Lane尽快运行对应 root 回调
普通异步更新选出 Default Lane按普通优先级排入 taskQueue
startTransition 更新标为可延后的 Transition Lane主线程繁忙时允许它等待或让位

Lane 决定“这轮 Render 包含哪些 update”；Scheduler 优先级决定“这轮 Render 的回调何时执行”。映射后，Scheduler 不再关心具体 Lane，也不读取 Fiber。

MessageChannel：暂停后，通知自己继续工作

React 的一次 Render 可能很长。如果一直占着主线程，浏览器就不能及时处理点击和绘制页面。因此 React 计算一小段后会先停下来，把主线程还给浏览器。

React 计算一小段 → 暂停 → 浏览器处理点击、绘制页面 → React 继续计算下一段

MessageChannel 只负责最后一步：安排一次稍后执行的回调，让 React 可以继续未完成的工作。

// 简化理解，不是完整源码
port1.onmessage = continueWork;  // 收到消息后，继续计算

if (这一小段时间用完了) {
  port2.postMessage(null);   // 通知：稍后继续
  return;                    // 现在先把主线程还给浏览器
}
一句话记忆：MessageChannel 不负责拆 Fiber，也不会让计算变快；它只是让“继续计算”排到稍后，使浏览器有机会在两段计算之间处理更紧急的事情。

这里使用 MessageChannel，是因为它能较快地安排下一次回调，又不像连续嵌套的 setTimeout 那样可能受到最小延迟限制。时间片是否结束，则由 Scheduler 的 shouldYield() 判断。

🔄 一轮的节奏(不停循环,直到没活):
① postMessage 排一个宏任务 → ② 浏览器:绘制 / 响应输入 → ③ onmessage 触发,干够 5ms → 还有活? 是 → 回到 ① ↺

每轮工作结束后，浏览器可以处理绘制和输入事件，这构成了时间切片的基础。

// ① 建通道:port1 收到消息就执行 performWorkUntilDeadline
const channel = new MessageChannel();
const port = channel.port2;
channel.port1.onmessage = performWorkUntilDeadline;

// ② 安排工作 = 让 port2 发一条消息 → 排进「宏任务」队列(不是立刻执行)
function requestHostCallback(cb){
  scheduledHostCallback = cb;
  port.postMessage(null);      // ★发消息 = 排一个宏任务
}

// ③ 每个宏任务里:干一片活;还有活就再发消息,排下一片
const performWorkUntilDeadline = () => {
  startTime = getCurrentTime();                          // 记本片起点(shouldYield 要用)
  const hasMore = scheduledHostCallback(true, startTime); // 干活(内部每个单元后看 shouldYield)
  if(hasMore) port.postMessage(null);              // 还有活 → 排下一个宏任务,续下一片
};

// 「这一片 5ms 到了没?」——工作循环每处理一个单元就问它一次
const frameInterval = 5;
function shouldYieldToHost(){
  return getCurrentTime() - startTime >= frameInterval;
}

📖 逐句拆解上面这段 MessageChannel 代码

这段代码藏了 Scheduler 最核心的工程决策。分层看。

先明确目标

React 想实现：干一片活（5ms）→ 让浏览器 paint + 处理输入 → 再干下一片。

JS 里"排队再执行"有好几种方案：

方案类型问题
setTimeout(fn, 0)宏任务❌ 嵌套 5 次以上被钳制到最小 4ms（HTML 规范）
Promise.resolve().then微任务❌ 微任务里浏览器不能 paint / 处理输入
requestAnimationFrame帧调度❌ 只在下一帧前（~16ms），太慢；tab 后台不触发
requestIdleCallback空闲时❌ 只在浏览器空闲时，不稳定
MessageChannel宏任务✅ 无钳制、稳定、快速

核心痛点：setTimeout 一秒最多 250 片，严重限速。MessageChannel 没这个限制，可以背靠背跑几百片。

🧮 展开：为什么 setTimeout "一秒最多 250 片"

数学：HTML 规范规定嵌套调用的 setTimeout(fn, 0) 从第 5 次开始会被强制成 setTimeout(fn, 4)（最小 4ms）。1000ms ÷ 4ms = 250 次触发/秒 —— 这就是"一秒 250 片"的由来。

"嵌套"是啥：不是函数嵌套调用，而是回调里再调 setTimeout 这种链式：

function tick(){
  // 干一片活
  setTimeout(tick, 0);   // 排下一片
}
setTimeout(tick, 0);

React 的时间切片就是这个模式。典型深度嵌套。

具体钳制时序：

第 1 次 setTimeout(fn, 0)  → 实际 0ms 后触发  ✅
第 2 次(回调里调)          → 0ms             ✅
第 3 次                    → 0ms             ✅
第 4 次                    → 0ms             ✅
第 5 次 ← 从这开始钳制!    → 强制 4ms        ⚠️
第 6 次                    → 4ms             ⚠️
...

为啥有 4ms 钳制？早期浏览器防止 setTimeout(fn, 0) 无限循环卡死浏览器给的下限，HTML5 就有，所有浏览器都遵守。

对 React 5ms 切片的具体影响：

setTimeout 版MessageChannel 版
每周期干 5ms + 强制等 4ms = 9ms干 5ms + 空档 <1ms = <6ms
一秒周期数1000/9 ≈ 1111000/6 ≈ 166+
一秒实际干活时长≈ 555ms（其余等钳制）≈ 830ms+（几乎没浪费）

实测代码（可以自己跑）：

// setTimeout 版
let cs = 0, ts = Date.now();
function loopS(){
  cs++;
  if (Date.now() - ts < 1000) setTimeout(loopS, 0);
  else console.log('setTimeout 一秒:', cs);
}
setTimeout(loopS, 0);

// MessageChannel 版
let cm = 0, tm = Date.now();
const ch = new MessageChannel();
ch.port1.onmessage = () => {
  cm++;
  if (Date.now() - tm < 1000) ch.port2.postMessage(null);
  else console.log('MessageChannel 一秒:', cm);
};
ch.port2.postMessage(null);

典型结果：

  • setTimeout ≈ 250 次（硬顶）
  • MessageChannel ≈ 10000+ 次（背靠背无钳制）
  • 差距 40 倍

🪜 爬楼梯类比

  • setTimeout 的楼梯：每层门口有强制休息室，规定至少歇 4 秒才能上下一层
  • MessageChannel 的楼梯：没这个规定，你想歇多久就多久（React 里就是"干完 5ms 立刻上下一层"）

关键理解：setTimeout 的 4ms 钳制不是"给浏览器 4ms 干别的事"，而是"强制你什么都别干" —— 这 4ms 里浏览器可能已经空闲了，但你没法立刻回来干活。MessageChannel 消除了这个人为等待，让"干活密度"最大化。

MessageChannel 是什么

原本是 Web Workers 通信 API，但可以在同一个页面里给自己发消息：

const channel = new MessageChannel();
channel.port1.onmessage = () => console.log('收到!');
channel.port2.postMessage(null);
console.log('当前同步代码');

// 输出:
// 当前同步代码
// 收到!

关键：postMessage() 不会立即触发 onmessage，而是排到宏任务队列。当前 JS + 微任务 + 浏览器 render + 输入事件 → 然后才轮到它。中间那段空档就是「让位给浏览器」的物理机制。

逐句读代码

① 建通道

const channel = new MessageChannel();
const port = channel.port2;
channel.port1.onmessage = performWorkUntilDeadline;

像建了一根管子：port2 塞字条，port1 读字条。塞进去 → 排一个宏任务 → 稍后触发读端。

② 排队入口

function requestHostCallback(cb){
  scheduledHostCallback = cb;
  port.postMessage(null);   // ★发消息 = 排一个宏任务
}

把要跑的活存到模块变量，然后塞一张空白字条 —— 排一个宏任务。null 是啥不重要，React 只利用"塞字条 → 排宏任务"这个副作用。

③ 干一片活

const performWorkUntilDeadline = () => {
  startTime = getCurrentTime();                          // (1)记起点
  const hasMore = scheduledHostCallback(true, startTime);  // (2)干活
  if(hasMore) port.postMessage(null);                    // (3)续片
};
  1. 记录起点 startTime → 供 shouldYield 判断"过了几毫秒"
  2. 干活 = 调 scheduledHostCallback（就是 Scheduler 的工作循环，遍历 task 最小堆）
  3. 续片 = 如果活没干完，postMessage(null) 再排一个宏任务

④ 判断该停了

const frameInterval = 5;
function shouldYieldToHost(){
  return getCurrentTime() - startTime >= frameInterval;
}

工作循环每处理完一个 task就问一次："距离 startTime 是不是过了 5ms？"过了就退出这一片。

🎬 一轮完整时间线

T=0    调 requestHostCallback(cb)
         cb 存入 scheduledHostCallback
         postMessage(null) → 排一个宏任务
         当前 JS 继续跑,直到本轮同步代码结束

T=1    ┌──────────────────────────────────────┐
       │ 微任务清空 → 浏览器 paint / 处理输入 │  ← 空档 1
       └──────────────────────────────────────┘

T=2    宏任务 tick → onmessage 触发
       performWorkUntilDeadline:
         startTime = T2
         scheduledHostCallback 循环:
           处理 task1 → shouldYield? (T2+0.5 vs T2+5) → 否
           处理 task2 → shouldYield? (T2+1.2 vs T2+5) → 否
           ...
           某次问: (T2+5.1 vs T2+5) → 是! → 退出
         hasMore = true → postMessage 排下一片

T=3    ┌──────────────────────────────────────┐
       │ 浏览器 paint / 处理输入              │  ← 空档 2
       └──────────────────────────────────────┘

T=4    下一个宏任务 tick → 同 T2 循环,再干 5ms

...    循环直到 hasMore = false

每两片之间的空档就是「让给浏览器」的时机 —— 这就是页面能保持流畅的物理原因。

前端类比：你可能写过

let index = 0;
function processChunk(){
  const start = Date.now();
  while (index < items.length && Date.now() - start < 5){
    process(items[index++]);
  }
  if (index < items.length) setTimeout(processChunk, 0);
  //                       ↑ ← 唯一区别在这里
}

几个易混点

问答
postMessage(null) 里的 null 有意义吗？没意义。React 只利用"post 一次就排一个宏任务"这个副作用
onmessage 触发时收到什么？也没意义。onmessage 只是"到时机了叫我"的通知
5ms 从哪来的？经验值。太短（1ms）切片开销高；太长（16ms）输入延迟明显。5ms 是甜蜜点
不支持 MessageChannel 咋办？fallback 到 setTimeout(fn, 0) —— 能用则用
一句话总结：MessageChannel = "无 4ms 钳制的 setTimeout(fn, 0)"。postMessage 排宏任务，浏览器在两个宏任务之间能 paint 和处理输入，shouldYieldToHost 用 startTime 判断"这片够 5ms 了吗"。循环 = 干一片 → 让浏览器一下 → 干一片 → ...直到活干完。

为什么比 setTimeout / rIC 精准

方案问题
setTimeout(fn,0)嵌套后被强制最小 4ms 延迟,累积浪费
微任务(Promise)绘制前就清空,阻塞渲染,拿不到帧间空档
requestIdleCallback触发不稳定、频率低、兼容差
MessageChannel无 4ms 钳制的宏任务,绘制/事件后尽快触发 → 卡在帧之间

🍽️ 展开：为什么 Promise 微任务会阻塞渲染

"绘制前就清空，阻塞渲染，拿不到帧间空档"这句藏了浏览器事件循环最关键的规则。展开讲。

浏览器一"帧"里发生什么

1. 取一个宏任务(setTimeout / MessageChannel / click 等)执行
2. 清空所有微任务(Promise.then / queueMicrotask 等)
3. 检查是否需要 rAF 回调
4. 需要则 layout / paint (★渲染画面)
5. rIC 回调(空闲时)
6. 回到 1

两个关键规则：

  • 每处理完一个宏任务，浏览器先把微任务清空，才 paint
  • 微任务里产生的新微任务会一直执行，直到队列真的空

用 Promise 做时间切片会怎样

function tick(){
  const start = Date.now();
  while (Date.now() - start < 5){
    doWork();
  }
  if (hasMore) Promise.resolve().then(tick);   // 排"下一片"
}
Promise.resolve().then(tick);

执行流程：

① 当前宏任务结束
② 浏览器进入"清空微任务"阶段
③ 执行 tick → 干 5ms 活 → Promise.resolve().then(tick) 又排一个微任务
④ 微任务队列还没空,继续执行下一个 tick
⑤ 干 5ms 活 → 又排一个
⑥ 继续 ...

问题：微任务队列永远清不空。每次执行都往队列里塞新的 → 浏览器永远走不到 paint 阶段 → 页面完全冻结。JS 在跑，但用户看不到任何更新。

"绘制前清空 / 阻塞渲染"具体是啥

  • 绘制前清空 = paint 之前必须先清空微任务队列（浏览器规则）
  • 阻塞渲染 = 微任务不停产生新微任务 → 队列不空 → paint 永远轮不到
  • 拿不到帧间空档 = 微任务之间不会插入 layout/paint/输入处理

对比：MessageChannel 宏任务版

① 取宏任务(tick 第1片) → 执行 → postMessage 排下一片
② 当前宏任务结束
③ 微任务清空(空)
④ ★ 浏览器 paint ← 画面能更新
⑤ 处理输入事件
⑥ 取下一个宏任务(tick 第2片) → 执行 → postMessage 排下一片
⑦ 又能 paint
...

每两片之间都有 paint 机会。用户看到的是页面持续更新，交互跟手。

🍳 厨房类比

  • 微任务模式：「我先把这盘菜炒完再看客人」—— 一旦开始就必须做完所有微任务，客人（浏览器 paint）得等
  • 宏任务模式：「我做完这盘就出门透透气」—— 每个宏任务结束都是「透气」时机，客人可以插进来

微任务是「贪婪」的（不给别人机会），宏任务是「礼让」的（每次给别人机会）。时间切片要求"每片之后都让出" —— 微任务的贪婪特性正好是它的死穴。

可实测验证

// ⚠️ 微任务版会让页面立刻冻结,别真跑或先保存工作
function tickMicro(){
  const start = Date.now();
  while (Date.now() - start < 5){}
  Promise.resolve().then(tickMicro);
}
Promise.resolve().then(tickMicro);   // 页面立刻卡死,直到 tab 被浏览器杀掉

// 宏任务版:页面流畅
function tickMacro(){
  const start = Date.now();
  while (Date.now() - start < 5){}
  setTimeout(tickMacro, 0);
}
setTimeout(tickMacro, 0);   // UI 保持响应
一句话总结："绘制前清空、阻塞渲染、拿不到帧间空档" = Promise 微任务永远排在 paint 之前，而且如果自我 chain 会填不满队列，导致 paint 永远轮不到。所以 React Scheduler 不能用 Promise，只能用宏任务 —— 不是选择题，是必答题。

Scheduler 内部怎样保存任务

到这里再看最小堆就容易了：堆里的一个 React 任务，通常是某个 root 的工作回调，不是一个 Fiber。taskQueue 保存已经可以运行的任务，按 expirationTime 排序；timerQueue 保存尚未到开始时间的延迟任务，按 startTime 排序。advanceTimers 会把已经到点的任务移入 taskQueue。

任务饿死兜底:过期升级

低优任务可能一直被高优抢占而"饿死"。防饿死机制:每个任务有 expirationTime = startTime + timeout。workLoop 里若 currentTask.expirationTime <= currentTime(已过期),即使该让位也强制同步执行(didUserCallbackTimeout=true),保证低优任务最终一定被执行。
