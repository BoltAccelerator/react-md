5 · UpdateQueue：更新如何入队

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

一个组件 = 一个 Fiber = 一个 UpdateQueue;setState 把更新排进队列,渲染时一起 reduce 出最终 state。

本章包括：① update 长啥样 ② 怎么入队(环状链表)③ 怎么算(processUpdateQueue)④ baseState/baseQueue(并发存档)。

每次状态更新都会进入对应的更新队列。不过，类组件/HostRoot 与 useState Hook 使用的队列字段并不相同。它们的共同目标都是：保存新更新、按插入顺序计算，并在跳过低优先级更新后保留重算所需的信息。

① 先区分类组件队列与 Hook 队列

// 一个 Update 对象(class 组件的形态;hook 版字段略有差异)
const update = {
  lane,              // 本次更新的优先级车道
  payload,           // setState 传的对象,或 (state)=>newState
  next,              // → 下一个 update(环状链表)
};

类组件与 HostRoot

fiber.updateQueue = {
  baseState,                  // 计算的「起点」state
  firstBaseUpdate,            // 基础队列的头
  lastBaseUpdate,             // 同一队列的尾
  shared: { pending: null },  // 新进入的环状队列
}

firstBaseUpdate 与 lastBaseUpdate 不是两个队列，而是同一条单向链表的头、尾两个指针。

useState / useReducer Hook

hook = {
  baseState,
  baseQueue,                  // 基础环状队列的尾指针
  queue: {
    pending: null             // 新进入的环状队列
  }
}

Hook 没有 firstBaseUpdate/lastBaseUpdate；它用一个 baseQueue 尾指针表示基础环状队列，队头同样是 baseQueue.next。

💡 为什么要同时保存 first 和 last？（链表基础）

updateQueue 是一条单向链表而不是数组。节点长这样：{ action, next }，节点之间靠 next 串起来 —— 你手上拿到哪个节点，就只能从它往后走，回不了头。

[u1] → [u2] → [u3] → null

类比 DOM：浏览器里 parent.firstChild 用来从头遍历子节点，parent.lastChild 用来 O(1) 找末尾；appendChild 是 O(1) 的原因，就是浏览器同时保存了 lastChild。React 的 firstBaseUpdate / lastBaseUpdate 是同一个思路。

操作只有 firstfirst + last
处理队列（从头往后 reduce）O(n) ✅O(n) ✅
追加新 updateO(n)——必须一路 node.next 走到末尾O(1)——last.next = new; last = new;
额外内存1 个指针2 个指针

多存一个尾指针的代价，换掉每次追加时对整条队列的遍历 —— 这是所有链表设计追加操作时的通用做法，不是 React 特有的。

② 新 update 先进入 pending 环状链表

下面先看类组件的 shared.pending；Hook 的 queue.pending 插入方式相同。多次更新形成环状链表，pending 指向最后一个节点，pending.next 指向第一个节点。

💡 一步步理解「环状链表 + 一个 pending 指针」

先回想上一节：普通单链表要 O(1) 追加，得同时存 first 和 last 两个指针。React 想再省一步 —— 用一个指针同时表达头和尾。

关键魔法：把链表的尾巴接回头，变成环：

从这样：  [u1] → [u2] → [u3] → null

变成这样：  [u1] → [u2] → [u3]
             ↑              │
             └──────────────┘
                 u3.next = u1

环成了以后，一个指针 pending 指着尾巴 u3 就够了：

pending          → u3   // 这是尾
pending.next     → u1   // 顺着尾的 next 就到头

很多人下意识觉得 pending 应该指第一个节点 —— 反了。它指最后一个，因为「尾 + 环」等价于「头 + 尾」两个信息。

插入过程分步演示

代码就三行（简化）：

function enqueueUpdate(queue, update){
  const pending = queue.pending;
  if(pending === null){
    update.next = update;              // 第一个,自己指自己
  } else {
    update.next = pending.next;        // 新节点.next = 旧的头
    pending.next = update;             // 旧的尾.next = 新节点
  }
  queue.pending = update;              // pending 前移到新的尾
}
  1. 第 1 次插入 u1（空环）：u1.next = u1，pending → u1。一个人的环，自己指自己。
  2. 第 2 次插入 u2：三步指针 —— ① u2.next = u1（新节点指向旧的头）② u1.next = u2（旧的尾指向新节点）③ pending 前移到 u2。此时 pending = u2（尾）、pending.next = u1（头）。
  3. 第 3 次插入 u3：同样三步，结果 pending = u3、pending.next = u1，环变成 u1 → u2 → u3 → u1 → ...

关键观察：不管链表已经多长，每次插入永远只改 3 个指针 —— O(1)。

和上一节的方案摆一起对比

方案找头找尾追加需要几个指针
普通链表，只有 firstfirst从头遍历O(n) ❌1
普通链表，first + lastfirstlastO(1) ✅2
环状链表，只存 pending（=尾）pending.nextpendingO(1) ✅1

环状链表把普通的双指针方案压缩成一个指针 —— 用「绕一圈能回到头」的性质，从「存两个位置」降级为「存一个位置」。

先把范围说准确：“pending 环转成普通链表”特指类组件与 HostRoot。Hook 的 queue.pending 也是环，但它会与同为环状的 baseQueue 合并，处理时仍可用“记住队头，绕一圈回到队头即结束”的方式遍历。

为什么同一批 update 要切换两种形态

因为这批 update 在进入 Render 前后承担的是两种完全不同的任务。可以把 pending 看成收件箱，把基础队列看成 Render 的工作清单：

🚚 用「搬家」类比想通

在旧房子（收集阶段）：东西不断打包成箱子往门口堆。你不需要「从第几箱开始拆」，也不关心「最后一箱是啥」，只关心「最新一箱放哪最方便下一箱堆」 —— 这就是环状链表用一个 pending 尾指针的语义。

到新房子（处理阶段）：搬家公司把所有箱子按顺序摆好，你从第一箱开始拆，拆一箱丢一箱，拆到没箱子就完事 —— 这就是普通链表 while(node !== null) 的语义。

在旧房子里想着「怎么拆」很傻（可能还要再打包），在新房子里想着「下一箱塞哪」也很傻（已经全摆好了）。中间那一次「断环」就是搬家公司帮你从旧房搬到新房的动作 —— 也就是稍后会讲的那 4 步指针操作。

读写分离视角

如果你写过数据库、消息队列或事件循环，这个模式很熟：

setState（写）        Render（读）
    │                      │
    ▼                      ▼
[环状 pending]  ──断环──►  [普通 baseQueue]
                   (Render 开始时)
  • 写侧：可能 1 次也可能 100 次，只要塞得进去 → 用环（一个指针搞定）
  • 读侧：一次性遍历、要能跳过低优 update 存 baseQueue → 用普通链表
  • 断环：Render 开始时把写侧攒的一批数据整体交接给读侧

为什么不用一种结构撑到底

假设 Render 阶段还继续用环状链表遍历：

let node = pending.next;   // 从头开始
const start = node;         // ★还得记住起点做哨兵
do {
  reduce(node);
  node = node.next;
} while(node !== start);    // 环上永远绕圈,得靠哨兵才停

遍历要多传一个「起点哨兵」，代码就复杂了；处理途中要跳过 update、重新拼一条 baseQueue，在环上改指针也更别扭。用 null 结尾的普通链表，while(node !== null) 一行就完事 —— 这是所有前端都写熟的模式。

反过来，如果收集也用普通链表，就变回上一节的问题：要么追加 O(n)、要么就得存 first+last 两个指针，比环更费。没有一种数据结构能同时把两种需求做到最优，所以 React 选择「收用环、读用链表」，中间做一次切换。

所处阶段主要操作适合的形态为什么
事件发生时：收集不断把新 update 追加到末尾尾指针表示的环只存 pending 一个指针，就能用 pending 找队尾、用 pending.next 找队头；每次追加都是 O(1)。此时不需要遍历或计算 state。
Render 开始后：处理从最早的 update 向后计算，并把跳过的 update 保存成新的基础队列类组件使用普通单链表从 firstBaseUpdate 一路走到 null 即结束；同时用 lastBaseUpdate 快速追加，适合顺序消费和重建基础队列。

环转普通链表时，具体只发生四件事

  1. 先保存队尾：last = shared.pending，例如得到 u3。
  2. 在断环前保存队头：first = last.next，因此仍能找到 u1。
  3. 断开队尾到队头的回边：last.next = null，于是 u1 → u2 → u3 → u1 变为 u1 → u2 → u3 → null。
  4. 接入工作清单：把 first/last 追加到类组件基础队列的 firstBaseUpdate/lastBaseUpdate 后面。

🎬 一步步看指针怎么变

起点：已经攒了三个 update 在 pending 环里，fiber 上还挂着一条已有的 baseQueue（比如上一轮跳过的低优 update b1 → b2）。

shared.pending  ─→ u3
                    │
   ┌────────────────┘
   ▼
  [u1] → [u2] → [u3]
   ↑              │
   └──────────────┘   ← u3.next = u1（环）

firstBaseUpdate ─→ [b1] → [b2] → null
lastBaseUpdate  ─→ [b2]

目标：把环上的 u1 → u2 → u3 接到 baseQueue 末尾，变成 b1 → b2 → u1 → u2 → u3 → null。

① last = shared.pending —— 拿临时变量记住尾巴 u3。没改任何指针。

last ─→ u3   （新变量，指向 u3）
环状结构不变

② first = last.next —— 通过 last.next 得到头（因为环上 u3.next = u1）。还是没动指针，只是记录。

last  ─→ u3
first ─→ u1   （通过 last.next 得到）

⚠️ 这一步为什么不能省？下一步就要断环了。断环后 u3 → u1 那条边没了，如果没提前把 u1 记下来，永远找不到 u1 —— u1/u2 会一起丢失。

③ last.next = null —— 真正的「断环」。把 u3 原本指向 u1 的那条回边切了。

[u1] → [u2] → [u3] → null    ← 已经是普通链表了!

last  ─→ u3
first ─→ u1

④ 挂到 baseQueue 末尾

if(lastBaseUpdate === null){
  firstBaseUpdate = first;         // baseQueue 空,直接接头
} else {
  lastBaseUpdate.next = first;     // b2.next = u1
}
lastBaseUpdate = last;             // 尾巴前移到 u3

结果：

firstBaseUpdate ─→ [b1] → [b2] → [u1] → [u2] → [u3] → null
lastBaseUpdate  ─→ [u3]

整合成一段可执行代码

if(shared.pending !== null){
  const last  = shared.pending;    // ① 记尾
  const first = last.next;         // ② 记头(必须在断环前)
  last.next = null;                // ③ 断环
  shared.pending = null;           // 清空收件箱

  // ④ 挂到 baseQueue 末尾
  if(lastBaseUpdate === null) firstBaseUpdate = first;
  else lastBaseUpdate.next = first;
  lastBaseUpdate = last;
}

1-3 步 = 断环（从环形变直链）；第 4 步 = 接入（拼到工作清单后面）。做完之后 Render 就能用 while(node !== null) 从头 reduce 到尾。

📌 展开讲第 ④ 步：「接入工作清单」到底做什么

先分清「字段」和「队列」

firstBaseUpdate / lastBaseUpdate 是 fiber.updateQueue 上的两个字段，值是"指向 update 节点的引用"。整条"基础队列"是从 firstBaseUpdate 顺着 .next 走到 null 得到的逻辑链，本身不是独立对象。

名字是什么类比
firstBaseUpdate字段，值是"第一个 update 节点"DOM 的 parent.firstChild
lastBaseUpdate字段，值是"最后一个 update 节点"DOM 的 parent.lastChild
基础队列 / baseQueue / 工作清单逻辑概念，一整条链DOM 的 parent.children
firstBaseUpdate ────┐
                    ▼
                   [b1] ───► [b2] ───► null
                              ▲
lastBaseUpdate  ──────────────┘

两个字段是「入口」和「出口」的书签，节点之间靠自己的 next 连接。

为什么「工作清单」可能已经有东西

baseQueue 是跟着 fiber 走、跨渲染持久化的。上一轮渲染跳过没处理的低优 update 会留在这里，等下一轮再算。举例：

第 1 轮渲染前 pending 里有:
  [高优 u1] → [低优 u2] → [高优 u3]   (环)

第 1 轮只算高优 → 处理完 u1、u3
低优 u2 不能丢!存进 baseQueue:
  firstBaseUpdate ─→ [u2]
  lastBaseUpdate  ─→ [u2]

进入第 2 轮时,baseQueue 里已经有 u2 —— 它是上一轮的「剩饭」。

所以第 ④ 步必须是「追加」，不是「替换」：新 setState 排在老剩饭后面，才能保证 update 处理顺序 = setState 调用顺序。

两种情况的具体动作

情况 A · baseQueue 是空的（首次渲染，或上轮全处理完）：

接入前:
  firstBaseUpdate = null
  lastBaseUpdate  = null
  新断出来的:  [u1] → [u2] → [u3] → null

代码: firstBaseUpdate = first;   // 头书签贴上 u1
      lastBaseUpdate  = last;    // 尾书签贴上 u3

接入后:
  firstBaseUpdate ────┐
                     ▼
                    [u1] → [u2] → [u3] → null
                                    ▲
  lastBaseUpdate  ──────────────────┘

情况 B · baseQueue 有内容（上轮跳过了 b1、b2）：

接入前:
  firstBaseUpdate ─→ [b1] → [b2] → null
  lastBaseUpdate  ─→ [b2]
  新断出来的:  [u1] → [u2] → [u3] → null

代码: lastBaseUpdate.next = first;   // ★真正改链:b2.next 从 null 变为 u1
      lastBaseUpdate      = last;    // 尾书签从 b2 移到 u3

接入后:
  firstBaseUpdate ─→ [b1] → [b2] → [u1] → [u2] → [u3] → null
                                                   ▲
  lastBaseUpdate  ─────────────────────────────────┘
关键观察：三条赋值语句里，只有 lastBaseUpdate.next = first 真的在改链表结构 —— 把 b2 原本的 .next = null 改成 .next = u1。另外两条（firstBaseUpdate = first 和 lastBaseUpdate = last）只是移动"书签"字段，链上节点本身没变。
为什么一定要先读 last.next 再断环？因为外部只保存了队尾 last，队头藏在 last.next 中。如果先执行 last.next = null，就会丢失队头入口，无法找到 u1。正确顺序是“先取队头，再断开回边”。

所以，环并不是“计算 update 更方便”，普通链表也不是“入队更快”。React 是在每个阶段选择更适合当下操作的表示：环负责高频收集和整批移交，普通链表负责有明确终点的顺序计算。

为什么收集阶段选择尾指针环

pending 可能连续插入很多次，但一轮 Render 通常只需接管这个入口。环状结构让收集阶段只保存一个 pending 指针，却能同时 O(1) 找到队尾和队头：

结构如何找队头如何找队尾尾部插入需要几个外部指针
普通单链表，只存 headhead必须从头遍历O(n)1
普通单链表，存 head + tailheadtailO(1)2
环状单链表，只存 tailtail.nexttailO(1)1
环的收益发生在入队阶段：假设 Render 前来了 100 个 update，React 会做 100 次 O(1) 入队；Render 接管时只做一次 O(1) 的断环。总成本仍是 O(n)，但不会出现“第 1 次找 0 个节点、第 2 次找 1 个节点……第 100 次找 99 个节点”的反复尾部遍历。
null 终点为什么适合类组件的处理过程？processUpdateQueue 不只是读取旧队列，还会在遇到低优先级 update 后构造一条新的基础队列。普通链表可以用 update !== null 控制遍历，并用新的 first/last 指针逐个追加需要保留的节点；无需额外保存“绕一圈后的停止哨兵”。

普通的 head + tail 双指针链表也能实现 O(1) 入队；环状链表不是唯一方案。React 的选择是用一个尾指针同时编码头和尾，并让 Hook 的 pending 环与 baseQueue 环可以通过少量指针修改完成合并。

连续插入 u1、u2、u3 后，指针怎样变化

最终只要记住两个入口：pending 是队尾 u3；pending.next 是队头 u1。遍历从 u1 开始，依次经过 u2、u3，再回到 u1 时结束。
function enqueueUpdate(fiber, update){
  const queue = fiber.updateQueue.shared;
  const pending = queue.pending;
  if(pending === null) update.next = update;    // 第一个:自己成环
  else { update.next = pending.next;             // 新节点接到头前
         pending.next = update; }
  queue.pending = update;   // pending 永远指向「最后进来的」
}
把源码代入“插入 u3”：
进入函数前：pending = u2，pending.next = u1。
① u3.next = pending.next，所以 u3.next = u1；
② pending.next = u3，所以 u2.next = u3；
③ queue.pending = u3，队尾指针从 u2 移到 u3。
旧的 u1.next = u2 不需要修改，最终得到 u1 → u2 → u3 → u1。

pending 还划出了一批更新的边界

shared.pending 不只是“指向队尾”。它还代表尚未被 Render 接管的整批新更新。Render 开始处理类组件队列时，会先取走这个指针并把 shared.pending 清空：

// Render 开始：把当前 pending 整批交给本轮计算
const lastPending = queue.shared.pending;  // u3（队尾）
queue.shared.pending = null;              // pending 槽位重新空出来

const firstPending = lastPending.next;      // u1（队头）
lastPending.next = null;                  // u1 → u2 → u3 → null

// 再把 firstPending...lastPending 接到基础队列尾部

u1 → u2 → u3 → u1

本轮持有 u1～u3

u4.next = u4
shared.pending → u4

这就是“批次边界”：React 只交换一个 pending 指针，就把 u1～u3 整批交给当前处理流程；之后到来的 u4 会在空出来的 pending 中重新成环，不会破坏已经拆出的链表。如果本轮处理结束前又发现了这批更新，React 可以继续把它接进来；否则由后续工作处理。

Hook 为什么也保留环：两个队列可以 O(1) 拼接

Hook 的 baseQueue 和 queue.pending 都是“尾指针 + 环”。假设旧基础队列是 b1 → b2 → b1，新 pending 是 p1 → p2 → p1。React 已经同时拿到了两个队尾，因此只要交换两条连接：

const baseFirst = baseLast.next;        // b1
const pendingFirst = pendingLast.next;  // p1

baseLast.next = pendingFirst;               // b2 → p1
pendingLast.next = baseFirst;               // p2 → b1，重新闭环
baseQueue = pendingLast;                     // 新队尾是 p2
合并后：b1 → b2 → p1 → p2 → b1。旧更新仍在前，新更新仍在后；无论两个环里各有多少节点，合并都不需要遍历，所以仍是 O(1)。这比“环能少存一个指针”更重要：它让 React 能快速接管、拼接和保留整批更新。

③ pending 与基础队列按什么顺序执行

pending 是“渲染开始前新收到的更新”，基础队列是“本次必须从哪里继续计算”。开始处理类组件队列时，React 会先把 pending 环拆开并追加到基础队列末尾，然后从头按插入顺序遍历。

④ 渲染时怎么算:从 baseState 逐个 reduce(processUpdateQueue)

渲染时 React 调用一个叫 processUpdateQueue 的函数来算最终 state:从 baseState(起点值)出发,按顺序把队列里每个 update 应用一遍(像滚雪球),得到本次的 memoizedState。这就是「多次 setState 被合并成一次结果」:

// processUpdateQueue 概念:从 baseState 开始,依次 apply 每个 update
let newState = queue.baseState;
for(const u of 队列里的 update){
  newState = reduce(newState, u);   // u 是对象则合并;是函数则 u(newState)
}
fiber.memoizedState = newState;      // 得到本次最终 state

所以 setCount(c=>c+1) 连写三次会累加(每个 update 拿到前一个的结果),而 setCount(count+1) 连写三次只 +1(闭包里 count 是同一个旧值)。

⑤ 为什么需要 baseState 和基础队列

先不要记字段名。只看同一个 count 按顺序收到的三次更新：

u1: c => c + 1（高优） → u2: c => c + 10（低优） → u3: c => c * 2（高优）

初始值是 0。React 先处理高优更新时，会按原顺序走一遍：

  1. 执行 u1：0 + 1 = 1。
  2. 暂时跳过 u2：它是低优更新。这一刻的值 1 被记为 baseState，意思是“下次补算要从这里开始”。
  3. 执行 u3：1 × 2 = 2，所以本次可以先显示 2。

问题是：以后补算 u2 时，不能只算 1 + 10 = 11。原始顺序是 u2 → u3，所以还要再算一次 u3，最终才是 (1 + 10) × 2 = 22。

基础队列就是“下次必须重放的那一段顺序”。从第一个被跳过的 u2 开始，React 保存 [u2, u3]。u3 本次虽然已经执行，仍要留一份副本，保证下次能恢复原来的计算顺序。

现在再记定义：baseState 是第一次跳过更新时已经算出的 state；基础队列是从该更新开始、下次需要按顺序重放的 update。这里分别是 1 和 [u2, u3]。

为什么 u3 的副本使用 NoLane？因为 u3 已经通过了本次的优先级检查。下次重放时，它必须跟在 u2 后面执行，不能再因 Lane 不匹配而被跳过；NoLane 在这里表示“这个副本重放时直接执行”。

“无关的更新”要分情况：另一个 useState 有自己的队列，不会混进来；但同一个 state 队列里的后续 update 都要保留顺序。React 不会猜测两个更新函数在业务语义上是否互不影响。

源码中，isSubsetOfLanes(renderLanes, update.lane) 决定本次是否跳过 update。类组件用 firstBaseUpdate/lastBaseUpdate 保存这段基础队列；State Hook 用 hook.baseQueue 保存。
