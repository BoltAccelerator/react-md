6 · Lane：选择本次更新

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
这一章建立 Lane 的完整工作模型

Lane 不只是“高、中、低”标签。它把每个 update 的优先级编码成二进制位，让 React 能在多种更新同时存在时，选择一批工作、跳过暂时不做的更新，并在之后按原顺序补算。

阅读顺序：为什么需要 → Lane/Lanes 是什么 → 如何分配与传播 → root 如何选一批 Lanes → Render 如何处理 → 挂起、唤醒、过期和提交后怎样变化。

💡 心智模型：Lane = 用二进制位当「优先级 tag」

在钻进 requestUpdateLane、pendingLanes 这些细节之前，先立一个通用直觉。假设你要设计一个数据结构，同时管理 31 种优先级的更新：合并、判断、取最高。两种做法对比：

方案合并判断包含取最高
数组: ['sync', 'transition']push + 去重includes(): O(n)排序或遍历
位掩码: 0b0101| 一条指令& 一条指令lanes & -lanes 两条指令

Render 阶段要做大量「这个 update 属于本轮吗」的判断，位运算比数组/字符串快几个数量级。这就是 React 用二进制位实现 Lane 的根本原因。

前端里的等价类比

  • Linux 权限 rwx：3 个权限各占 1 位，chmod 7 就是 0b111 —— 三个权限都开。合并、检查全靠位运算。
  • Tailwind class：class="bg-red-500 text-white p-4" 每个 class 独立开关，任意组合。你不会为「红底白字」发明新 class。Lane 也是「独立开关的组合」。

后面看到的 SyncLane = 0b0001、lanes |= Sync、lanes & renderLanes —— 这些都是这个心智模型的应用。

先回答：React 为什么需要 Lane

同一个 root 上可以同时存在不同来源、不同紧急程度的更新。例如用户正在输入文字，页面还要重新筛选一个大列表：

输入框更新用户正在等待反馈，应尽快显示。
列表筛选更新计算量大，可以放进 startTransition 延后处理。

如果 React 只用普通 FIFO 队列，就只有两个不理想的选择：严格按顺序执行，可能让大列表阻塞输入；或者粗暴地把低优更新挪走，又容易破坏同一 state 队列的计算顺序。Lane 的作用是：保留 update 的原始队列顺序，同时额外标记“这一轮要不要处理它”。

核心模型：UpdateQueue 保存“更新按什么顺序发生”；Lane 决定“本轮选择其中哪些更新”；baseQueue 保存“本轮跳过、以后必须按原顺序重放的更新”。三者配合，React 才能既优先响应紧急工作，又不算错最终 state。

Lane、Lanes 与 renderLanes 分别是什么

名称含义典型位置
Lane一个二进制位，通常表示一个 update 所属的优先级通道update.lane
Lanes多个 Lane 按位 OR 得到的集合，不是一种新的优先级fiber.lanes、root.pendingLanes
renderLanes当前 Render 选择处理的 Lane 集合，可能只有一个 bit，也可能包含多个Render 工作循环与 update 队列处理函数的参数
Lane 不是线程、任务或 state。它本质上是数字中的一个 bit。React 用位运算快速完成“加入集合、判断是否包含、合并、移除和寻找最高优先级组”。

第一步：创建 update 时分配 Lane

调用 setState 后，React 创建 update，并通过 requestUpdateLane() 根据当前上下文决定它属于哪类 Lane。常见来源如下：

更新来源常见 Lane含义
点击、键盘输入等离散交互SyncLane用户正在等待结果，通常需要尽快处理
连续输入，例如拖动、滚动InputContinuousLane紧急程度较高，但可以与离散同步更新区分
普通更新DefaultLane常规优先级工作
startTransition 内的更新某个 TransitionLane允许等待、暂停或被更紧急工作抢占
Suspense 数据恢复后的重试某个 RetryLane重新尝试之前挂起的边界
一次更新链的起点：事件/Transition 上下文 → requestUpdateLane() → update.lane → update 进入 UpdateQueue。Lane 只描述这条更新如何被调度，不改变它在队列中的先后位置。

📍 一次 setState 后，Lane 到底存在哪几个地方？

很多人以为 update.lane 就完了。其实一次 setState 会把同一位 lane bit 写到 4 个不同字段 —— 每一处解决不同问题。先建立这张全景图，后面的「冒泡」和「bailout」才有依托：

setState(SyncLane) 触发后:

  update.lane        = SyncLane   ← ① 单个 update 的标签
  Button.lanes       = SyncLane   ← ② 我自己的待办
  Main.childLanes    = SyncLane   ← ③ 我子树的待办(冒泡)
  App.childLanes     = SyncLane   ← ③ 我子树的待办(冒泡)
  root.pendingLanes  = SyncLane   ← ④ 全局待办

同一位 bit，4 个视角：「自己」/「我」/「我们家」/「整栋楼」。

字段回答的问题谁用到它
update.lane这一个 update 属于哪条 lane？Render 处理 UpdateQueue 时用来筛选
fiber.lanes这个 fiber 自己有哪些 lane 待处理？Render 判断「我这里还有活吗」
ancestor.childLanes我的子树里有哪些 lane 待处理？Render bailout 整棵子树（性能关键）
root.pendingLanes整棵应用还有哪些 lane 待处理？Scheduler 决定下一批干哪些活

Trello 类比一秒理解

Trello Board (Root)
  └─ Trello List (Fiber)
       └─ Trello Card (Update)
            └─ 标签: 「紧急」  ← 这是 update.lane
  • Update 上的标签（update.lane）：告诉你这一张卡是什么优先级
  • List 头上的汇总（fiber.lanes + childLanes）：这个列有哪些优先级的卡
  • Board 头上的汇总（root.pendingLanes）：整个看板有哪些优先级的卡

每一层维护自己的「标签汇总」，是为了能在不打开子层的情况下，快速判断「我下面有没有紧急活」 —— 这就是下一节讲的冒泡和 render 阶段的 bailout 依赖的基础。

第二步：Lane 从更新节点传播到 root

只有 Button 知道自己更新还不够，负责调度的是根节点。React 必须沿 Fiber 的 return 指针向父级走，让 root 知道“Button 这条分支有一个 Sync 更新”。源码把这个向上传递过程叫作冒泡。

🔢 先看懂 lanes |= Sync 这种写法

下面的图里会频繁看到 |=、& 这种位运算。跟平时写的 += 是同一种「复合赋值」，只是运算符换成了位运算：

count += 1        // 等价于 count = count + 1
lanes |= Sync     // 等价于 lanes = lanes | Sync

| 是按位 OR（不是 ||！那是逻辑或，完全不同）。一位一位比较，任何一边是 1 结果就是 1：

  0100    // Button 已经有 Default
| 0001    // 又来一个 Sync
------
  0101    // 结果:同时含 Default 和 Sync

用前端熟的 Set 做类比

lanes |= Sync 语义上完全等价于 lanes.add(Sync) —— 往一个位集合里加一个元素，重复加没关系（幂等）。整套对照：

位运算Set 等价做什么
a | bnew Set([...a, ...b])并集（返回新值）
a |= ba.add(b)把 b 加入 a（就地）
a & b[...a].filter(x => b.has(x))交集
(a & wanted) !== 0a.has(wanted)判断包含
a & ~ba.delete(b)从 a 里移除 b
为什么用 |= 不用 =？Button.lanes 之前可能已经有别的 lane。Button.lanes = SyncLane 会把原有的全冲掉；Button.lanes |= SyncLane 才是「加进去」。这跟 list.push(x) vs list = [x] 是同一个道理。
Button 产生 Sync 更新后，各层分别记录什么
Root：pendingLanes |= Sync
整棵应用待处理的 Lane 中有 Sync
↑ 沿 return 指针向父级登记
App：childLanes |= Sync
App 自己没更新，但它的子树中有
↑
Main：childLanes |= Sync
Main 自己没更新，但它的子树中有
↑
Button：lanes |= Sync
更新就发生在 Button 自己身上

三个字段不要混：Button.lanes 表示“我自己有更新”；Main/App.childLanes 表示“我的后代有更新”；root.pendingLanes 表示“整棵应用还有哪些 Lane 没处理”。

为什么祖先要记录 childLanes？Render 从 root 往下走时，可以据此找到 Button 所在的分支。某个分支的 childLanes 不包含本次 Lane，就说明下面没有相关更新，整棵子树可以跳过。

root 不只保存 pendingLanes

更新到达 root 后，会进入多个 Lane 集合中的一种状态。理解这些字段，才能看懂 getNextLanes 为什么有时选择、有时跳过某些工作：

root 字段表示什么什么时候变化
pendingLanes已经到达 root、但还没有全部完成的 Lane新 update 到来时加入；成功提交并确认完成后移除
suspendedLanes本次尝试时因数据等原因挂起的 LaneRender 遇到 Suspense 后标记
pingedLanes之前挂起，但依赖的数据已经就绪，可以重试的 LanePromise resolve 后由 ping 标记
expiredLanes等待太久、需要防止继续饿死的 Lane超过过期时间后标记，后续提高完成紧迫性
entangledLanes与其他 Lane 有共同完成关系的 Lane相关更新不应独立提交中间状态时建立关联

🎯 关键理解：xxxLanes 不是不同的队列

上面这几个字段容易让人误以为 root 上挂了 5 个平行的队列。不是的 —— 它们都是位掩码（都是数字），代表从不同视角看同一批位。就像你 Trello 里的一张卡可以同时被打上「pending + expired」两个标签。

pendingLanes    = 0b1111   // 有 4 种 lane 等待
suspendedLanes  = 0b0010   // 其中第 1 位挂起了(等数据)
pingedLanes     = 0b0010   // 同一位数据 ready 了,可以重试
expiredLanes    = 0b0001   // 第 0 位等太久要保底

「真正新鲜可干的 lane」就是这几个掩码之间的位运算组合：

const unblockedLanes = pendingLanes & ~suspendedLanes;   // 排掉挂起的
const nextLanes     = unblockedLanes | pingedLanes;      // 加回可重试的

这也是为什么 getNextLanes 能在几条位运算里就把复杂的调度状态收敛出来 —— 每一步都是集合运算。

🎬 用一个 Suspense 时间线走完 5 个字段

上面 5 个字段抽象讲很难懂。用一个真实场景走一遍就清晰。想象你在做 tab 切换，配了 Suspense：

<button onClick={() => setTab('profile')}>Profile</button>
<Suspense fallback={<Skeleton/>}>
  {tab === 'profile' && <UserProfile />}
</Suspense>

function UserProfile(){
  const user = use(fetchUser());   // ★数据未就绪时会 throw Promise
  return <div>{user.name}</div>;
}

t=0 · 用户点击 button

setTab('profile') → 产生 update, lane = SyncLane

pendingLanes    = 0b0001   ← 「待办清单里加了一件事」
suspendedLanes  = 0b0000
pingedLanes     = 0b0000

t=1 · Render 到 UserProfile，数据还没就绪

use() 抛出 Promise，Render 中断：

pendingLanes    = 0b0001   ← 事还在待办里
suspendedLanes  = 0b0001   ← ★被打上「卡住,等 fetchUser」标签
pingedLanes     = 0b0000

此时不能提交（界面坏的），显示 <Skeleton/>。SyncLane 从 pendingLanes 里没被清掉 —— 事没干完。

t=2 · Promise resolve，ping 信号到达

pendingLanes    = 0b0001
suspendedLanes  = 0b0001   ← 还标着「卡住」
pingedLanes     = 0b0001   ← ★同一位被标为「外部好了,能重试」

同一位 SyncLane 现在同时挂着 3 个标签：pending + suspended + pinged。这就是「多视角看同一批位」的意思。

t=3 · 重新调度，Render 成功，Commit

getNextLanes 通过位运算算出这次能跑什么：

const canRun = (pendingLanes & ~suspendedLanes) | pingedLanes;
//        = (0b0001 & 0b1110) | 0b0001
//        = 0b0000 | 0b0001
//        = 0b0001   ← SyncLane 又能跑了

这次 use() 拿到真数据 → commit，清除标签：

pendingLanes    = 0b0000   ← 干完了,全清
suspendedLanes  = 0b0000
pingedLanes     = 0b0000

再看 expiredLanes：防止低优被饿死

你用 startTransition 更新大列表，用户不断打字触发高优 Sync，Transition 一直被抢占：

t=0    startTransition(() => setBigList) → pendingLanes |= TransitionLane
t=0.5  用户打字,Sync 抢占,Transition 被丢弃重来
t=1    用户又打字...
...
t=5    5 秒过去了,Transition 一次都没渲完

React 察觉:太久了,再拖就饿死
  expiredLanes |= TransitionLane   ← ★「别再让它饿死!」

下一轮 Scheduler 看到 expired，强制同步做完这个 lane（不再允许被抢占）。这是兜底机制，保证低优最终一定会执行。

再看 entangledLanes：必须一起处理

最典型场景：useSyncExternalStore 保证不同组件读同一外部 store 时不撕裂，或同一 state 上 Sync + Transition 两次 setState 有依赖关系。React 用 entangle 说：这几条 lane 缠在一起，处理其中一条时必须把其它也一起处理，避免中间状态出错。

root.entangledLanes = Sync | Transition   ← 这两位缠住

// getNextLanes 内部逻辑:
if(selected & entangledLanes){
  selected |= entangledLanes   // 强制把关联的都拉进本轮
}

普通开发者接触这个字段不多，通常由 React 内部或状态库使用。理解「必须绑在一起」就够了。

字段生命周期速查

字段什么时候被打上什么时候被清除
pendingLanes新 setState 到达 root该 lane commit 成功后
suspendedLanesRender 中遇到 throw promisecommit 成功后；或 pinged 时「覆盖」
pingedLanesPromise resolve 触发 ping下轮 render 后清
expiredLanes某 lane 停留超过阈值commit 后清
entangledLanes特殊 API 触发（如 useSyncExternalStore）commit 后清
本节小结：5 个字段都是「给同一批 lane 打不同状态标签」—— pending 没干完 / suspended 等外部 / pinged 外部好了 / expired 拖太久 / entangled 绑一起做。getNextLanes 用几句位运算把这些标签综合起来，选出「这一轮该做哪些」。

第三步：getNextLanes 选择下一批工作

getNextLanes(root) 的目标不是“永远取最高的一个 bit”，而是从 root 的各种 Lane 状态中选出下一批值得执行、能够执行，并且适合一起执行的工作。可以按下面的顺序理解：

  1. 没有 pending 工作：返回 NoLanes。
  2. 先考虑非 Idle 工作：普通交互和页面更新完成前，不急着处理 Idle。
  3. 优先选择未挂起的 Lane：从 pendingLanes 中排除仍在 suspended 且尚未 ping 的部分。
  4. 没有新鲜可执行工作时，再看 pinged：数据已经就绪的挂起任务可以重试。
  5. 从候选中取最高优先级组：得到的可能是一组 Lane，而不一定只有一个 bit。
  6. 与正在进行的 WIP 比较：新 Lane 没有更高优先级时，React 通常继续已有工作，避免白白丢掉进度。
  7. 计算共同完成关系：进入 Render 时再纳入需要一起处理的 entangled Lane。
“选最高优先级”不等于“只选一个 Lane”。React 先用最高 bit 判断当前属于哪个优先级组，再返回这个组中适合共同处理的 Lanes；之后还可能考虑 entanglement 或特殊同步刷新。

哪些 Lane 会一起处理

情况选择结果设计目的
同一优先级组中的多个 Lane可以作为一批选择减少重复遍历，并保持同组工作的协调
高优 Lane 与普通低优 Lane 同时等待通常先处理高优组避免低优大任务延迟用户输入等紧急反馈
彼此 entangled 的 Lane有效 Render 范围会纳入关联 Lane避免提交这些更新之间不该出现的中间状态
特殊强制同步刷新可以把指定的额外 Lane 一起同步刷新满足历史导航、滚动恢复等必须同步完成的宿主场景

一个完整选择示例

假设大列表的 Transition 仍在等待，此时用户又输入一个字符，root 变成：

root.pendingLanes = Sync | Transition
renderLanes = Sync
本次 Render：先处理 Sync update；Transition update 留在基础队列与 root 的 pendingLanes 中，之后再处理。

这里分开不是因为一次 Render 技术上只能处理一个 Lane，而是一次调度决策：假设 Sync 只需 2ms，Transition 的大列表需要 80ms；若总是合并，用户可能要等完整计算结束才看到输入结果。若两个 Lane 因共同完成关系被 entangle，或进入特殊强制刷新路径，React 也能够让它们处于同一批有效 Render 范围。

第四步：Render 用 Lanes 筛选 update 与子树

Render 遍历 update 队列时，不是比较“它是不是唯一被选中的 Lane”，而是检查 update.lane 是否属于 renderLanes 这个集合：

if (isSubsetOfLanes(renderLanes, update.lane)) {
  state = applyUpdate(state, update); // 本轮包含它：执行
} else {
  cloneToBaseQueue(update);          // 本轮不包含：保留以后重放
}
因此一次处理多个 Lane 时不需要另一套算法：renderLanes 中有几个 bit，就能匹配几类 update。Sync 与 Transition 若都在集合中，两类 update 都通过检查；若集合里只有 Sync，Transition 就被跳过并保留。

Lane 还用于跳过整棵子树。若某个 Fiber 自己的 lanes 和后代的 childLanes 都不与本轮有效 Lanes 相交，beginWork 可以直接 bailout，不再执行该分支的组件和 Diff。

Lane 只负责表达“这轮包含哪些更新”以及它们的优先级关系。Lane 本身不会暂停 JavaScript，也不会保存或丢弃 WIP；取消旧任务、决定续算还是重算，属于下一章 Scheduler 与 Fiber 工作循环的配合。

二进制位到底怎样表示一组 Lane

先只用 4 位演示。每一种 Lane 独占一位；1 表示“集合中有这种 Lane”，0 表示“没有”。

Lane二进制读法
SyncLane0001第 1 位是 1
DefaultLane0100第 3 位是 1
TransitionLane1000第 4 位是 1

操作一：用 OR 把 Lane 加进 pendingLanes

root 已经等待一个 Transition，又来了一个 Sync。逐位执行 OR：任意一边是 1，结果就是 1。

  1000   // Transition
| 0001   // Sync
------
  1001   // pendingLanes：两种更新都还没处理

因此 root.pendingLanes = 1001 不是一种新优先级，而是一个集合：里面同时有 Sync 和 Transition。

操作二：用 AND 判断本轮是否包含某个 update

假设本轮只选择 Sync，所以 renderLanes = 0001：

  0001   // renderLanes
& 0001   // Sync update.lane
------
  0001   // 非 0：本轮处理 Sync update

  0001   // renderLanes
& 1000   // Transition update.lane
------
  0000   // 等于 0：本轮跳过 Transition update

操作三：从集合中取出最高优先级

React 把更低的二进制位定义为更高优先级，因此 getHighestPriorityLane(1001) 会取出最右侧的 1，得到 0001，也就是 Sync。源码常见写法是 lanes & -lanes；现阶段只需记住它的结果是“保留最右侧的 1”。

const nextLane = pendingLanes & -pendingLanes;
// pendingLanes = 1001 → nextLane = 0001 → SyncLane

🔍 拆解 lanes & -lanes：为什么它就是「取最高优先级」

这个技巧看起来神秘，其实靠的是补码性质。以 lanes = 0b1010（十进制 10）为例：

// -lanes 的算法:按位取反,再 + 1(二进制补码定义)
lanes  = 0b0000...1010
~lanes = 0b1111...0101
+1     = 0b1111...0110    // 这就是 -lanes

// 现在做 lanes & -lanes:
   0b0000...1010
& 0b1111...0110
   -------------
   0b0000...0010    // 只保留了最右侧的那位 1

数学性质：一个数和它的相反数按位 AND，永远只留最右侧那位 1（也叫 lowest set bit）。

所以 React 选择「低位 = 高优先级」 —— 是为了让这个位技巧直接等于「取最高优先级」。设计上一环扣一环。

如果反过来定义「高位 = 高优先级」，取最高就得循环找最左边的 1，代码更慢也更长。

三个运算对应三个问题：| 把 Lane 加入集合；& 判断集合是否包含某 Lane；lanes & -lanes 取出最高优先级 Lane。真实源码位数更多，但读法完全相同。
